# Known-weak points (detail)

Referenced from `../SKILL.md`. State current as of 2026-07-05.

1. **Routing truth still lives on 3 surfaces, but the third is now machine-readable and hook-enforced.** `settings.json` `"model": "opus"` (session default, tier alias; the earlier pinned-`claude-fable-5[1m]` state was fixed by US-002/#60, merged as `a1a8a72`/PR #68), the engine's cost-profile prose table (`commands/_council-engine.md:97-105`, per-spawn-site tier by profile), and `skills/council/model-routing.json` (spec_version `2.0`, US-005/#63/F5): `tiers` (the one sanctioned home for concrete `claude-*` IDs), `profiles` (`max-plan`/`api-billed`), 13 `spawn_sites` each carrying a tier alias per profile, and a structural `egress_policy` for OpenRouter. A new hard pre-commit hook, `pipeline/hooks/check_model_routing.py`, rejects pinned `claude-*` IDs in `spawn_sites` slots (R1) and checks profile completeness, egress-policy presence, tier resolution, and the fail-soft fallback (R2-R5). The `classification`/`scout-research`/`scoring` OpenRouter task map and `lenses: {}` still exist as legacy keys (`tasks`, `lenses`, `defaults`) the resolver expects, but they are no longer the whole file. PR #72 ("feat: unified two-account routing table (US-005 / F5)") merged to main as `92e95d4` and issue #63 closed (verified 2026-07-06). The remaining weakness is that no runtime consumer loads this file yet: the engine prose table and settings.json still state their values independently, so the surfaces can drift until a loader ships.

2. **Session state on 3 unsynced surfaces.** Observed live mid-campaign: `.claude/council/index.json` recorded `"contract_issue": null` for the then-active session even though `issues.md` had real GitHub issues #58-#66 filed against it; the field only got backfilled by the manual closeout commit (`74e34c5`, 2026-07-06), not by the issue export itself. `session.md`, `index.json`, and `~/.claude/council/global-registry.json` remain three independent write targets with no reconciliation job; expect the same lag in any future session.

3. **Private-repo coupling.** Historically 5 telemetry hook events (`SessionStart`/`PostToolUse`/`PostToolUseFailure`/`Stop`/`SessionEnd`) shelled directly to `$HOME/Development/my-claude-setup-private/hooks/session_telemetry.py`; a missing private repo meant every tool call spawned a failing python3. Fixed by US-001/#59 (PR #67, `c562ab8`): all 5 events now call `hooks/telemetry-dispatch.sh` in this repo, which fail-softs to `exit 0` if the private hook is absent. US-007/#65 landed via PR #73 (2026-07-06): ECE, resume-tailor, and soc-security were already extracted as gitignored symlinks into `my-claude-setup-private`; docx-to-pdf was the last tracked suite and moved (private-repo commit `d93d62c`); this repo's README carries the Extracted Suites manifest and restore path. Residual coupling: the symlinks resolve only on machines where the private repo's `install.sh` has run.

4. **Registry telemetry only durable since today.** `skills/council/registry.json` committed HEAD previously showed 0/67 real uses (every nonzero count was uncommitted session-local writes); as of commit `dc44611` the committed file shows 18 uses with `last_updated: 2026-07-02`. One day of durability is not the 2-4 week observation window the design doc requires before any usage-based pruning (`.claude/council/sessions/claude-config-model-optimization-20260702-0003/design.md`).

5. **`/careful` and `/freeze` are unwired.** `hooks/careful-mode.sh`, `hooks/freeze-mode.sh`, and `hooks/skill-telemetry.sh` each expose a `check` entrypoint meant to run as a blocking `PreToolUse` hook, but none is registered in `settings.json`, `hooks.json`, or `.claude/settings.local.json` (verified by grep, no matches). Even if wired, they read `TOOL_NAME`/`TOOL_INPUT` from environment variables (`hooks/careful-mode.sh:25-26`), but the actual hook contract delivers tool info as a stdin JSON payload (as `acceptance-gate.sh` correctly does). So `/careful enable` and `/freeze enable` currently protect nothing.

6. **Spec-promised hooks that do not exist.** `pipeline/specs/SKILL-SECURITY-SPEC.md` documents `check_security.py` as a to-be-added pre-commit hook (§7.1, with an explicit TODO checkbox), and `pipeline/specs/SKILL-TRIGGER-RELIABILITY-SPEC.md` documents `check_triggers.py` (§6.1) the same way. Neither file exists under `pipeline/hooks/` (only `check_commit_msg.py`, `check_context_load.py`, `check_frontmatter.py`, `check_isolation.py`, `check_prose.py`, `check_references.py`, `check_token_budget.py`, `_utils.py` do).

7. **Governance validators cover `^skills/` almost exclusively; CI runs none of them.** As of 2026-07-05, six of the nine local hooks in `.pre-commit-config.yaml` still gate on `files: ^skills/.*\.md$` (`skill-frontmatter`, `skill-references`, `skill-isolation`, `skill-context-load`, `skill-token-budget`, `skill-prose-check`). Two narrow, file-specific exceptions now exist: `settings-schema` (`files: ^settings\.json$`, entirely outside `skills/`, added with US-006/#64/PR #71) and `model-routing` (`files: ^skills/council/model-routing\.json$`, under `skills/` but not `.md`, added with US-005/#63/PR #72). Both are single-file guards, not a broadening of coverage: `agents/`, `commands/`, `pipeline/`, and `hooks/` remain entirely outside every governance hook's glob. `.github/workflows/validate.yml` runs `bash -n`, `python3 -m json.tool`, an installer smoke test, and gitleaks; it still invokes neither `pre-commit` nor any `pipeline/hooks/check_*.py` script directly, so a contributor who never runs `pre-commit run --all-files` locally can merge a violation of any of these nine hooks (including the two new ones) through CI with nothing catching it.
