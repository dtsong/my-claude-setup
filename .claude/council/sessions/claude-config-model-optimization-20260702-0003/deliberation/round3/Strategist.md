# Strategist Final Position — Claude Config & Model-Routing Optimization

**Revised recommendation:** Ship an MVP of six low-effort, high-frequency items in one `/ship` PR: F1 (alias fix + 1M conflict), F2 (permissions rewrite), F4 (stale ID refresh), F5 (unified two-account routing doc), **F3a** (fail-soft telemetry wrapper + remove private path from public settings.json), and **F7** (extract dormant suites now). None of these change live model behavior, so the MVP ships ungated. Gate every *activation* (first `routed_consult` caller, `/ship` or council-profile model flips, Phase 2 relay, prune decisions) behind an eval, staged: a 12-case smoke set (new **F11**) in v1.1, the full 38-case harness as the entry criterion for Phase 2 and pruning. OpenRouter stays Pattern B (client-side, API-account-only) on the Max plan; Pattern A relay is a gated trial, not a launch item.

**Concessions made and why:**

1. **Split F3 (from Operator).** I originally scoped F3 as one M-sized v1.1 fill-in. Operator correctly showed the ship-blocking half is XS: five string edits swapping the bare `python3 $HOME/.../private/hooks/...` call for a fail-soft dispatcher wrapper, plus a ~5-line existence guard. That removes break-on-clone, unblocks the API work account, and takes the private path out of public `settings.json`. **F3a re-scores to RICE ~1350 and joins the MVP**; my own v1 routing metrics and the evidence-based prune both depend on this telemetry pipeline surviving, so the coupling forces it earlier. The M-sized hardening (per-call interpreter perf, event batching, alert tuning) stays v1.1 as **F3b**. Operator overstates blast radius on the Max account (the hook already fails soft in practice here), but wins decisively on effort and coupling.

2. **Eval-before-activation, and a new F11 (from Oracle).** I conceded the sequencing symmetry: because F1/F2/F4/F5/F3a change no *live* model behavior and F4's stale IDs have zero callers, the MVP genuinely ships without a harness gate. In return I adopted Oracle's activation-boundary gate. **New F11 (12-case smoke eval, RICE ~84) enters v1.1** as the acceptance criterion for the first `routed_consult` caller and any live model flip. The full 38-case harness slides to v1.1+ and becomes the *entry gate* for F6 (Phase 2 relay) and F7-adjacent prune evidence. This is the right split: gate what changes behavior, not what corrects config.

3. **Extract dormant suites now, not v1.1 (from Craftsman/Scout).** I originally parked F7 as a fill-in. Craftsman and Scout both verified the registry has zero committed uses (all 10 nonzero entries are this session's uncommitted writes), so extraction cannot wait on usage data and does not need it. Extraction is a pure workload judgment (ECE/resume/docx/soc are off your TS-React / data-ML / meta-repo path) and is independent of the tracking question. **F7 moves into the MVP PR.**

4. **Haiku-in-family Max-plan routing (from Oracle/Skeptic/Scout).** Adopted into F5: on the Max plan, cheap-task lanes route to Haiku 4.5 (in-family, rate-limit-efficient, zero egress) rather than OpenRouter. OpenRouter cheap routing is an API-account optimization only.

5. **OpenRouter Pattern A downgraded to a gated trial (from Skeptic/Scout).** I keep Phase 2 relay out of this PRD (Won't-now), but concede the trust-boundary framing: if Pattern A is ever trialed it must sit behind a send-allowlist (no file contents/secrets), ZDR + no-training flags, provider allowlist, an actual-vendor-model audit log, and a kill switch, and must prove measurable lens diversity or collapse to Pattern B. This is a v1.1+ decision gated on the 38-case harness.

**Non-negotiables:**
- **MVP = F1, F2, F4, F5, F3a, F7** in one PR. All are XS/S/M, none flip live model behavior, all ship ungated. This is the irreducible high-value-per-effort core.
- **F6 (Phase 2 relay) and F8 (speculative data/ML skills) stay out of this PRD.** F6's target `council-deliberate.js` does not exist locally (XL, not XL-minus); F8 has unvalidated demand (TS/React already covered by vercel-react-best-practices + web-design-guidelines + frontend-qa).
- **No live model flip ships without its eval gate.** F11 (12-case) gates first `routed_consult` activation; 38-case gates Phase 2 and any prune-by-evidence.
- **Cut scope, not quality.** Config correctness (F1) and fail-soft safety (F3a) are quality floors, not optional polish.
